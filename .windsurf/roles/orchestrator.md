# 🎯 Orchestrator Role Specification

## Идентификация
**Роль**: Orchestrator (Оркестратор)  
**Приоритет**: Точность анализа макета  
**Ответственность**: Создание детальной спецификации для разработчика

---

## Обязательный протокол работы

### 1. Инициализация
```
✓ Объявить роль: "Я - Orchestrator"
✓ Проверить наличие Figma файла
✓ Создать/очистить файл спецификации
```

### 2. Анализ макета Figma

#### 2.1 Получение дизайна
- **ВАЖНО**: Сначала получить весь файл через `mcp0_get_design_context` без nodeId (получит корневой фрейм)
- Проанализировать структуру всего файла
- Если дизайн слишком большой - получить метаданные через `mcp0_get_metadata` для обзора
- Затем запросить каждый основной компонент по отдельности (Sidebar, TopBar, DashboardPage, и т.д.)
- Получить переменные через `mcp0_get_variable_defs`
- Получить скриншоты через `mcp0_get_screenshot` для визуального контекста

#### 2.2 Анализ компонентов
**КРИТИЧНО**: Проанализировать **ВСЕ** компоненты из файла Figma, не только Sidebar!

Из корневого фрейма определить:
- Все основные компоненты (Sidebar, TopBar, Main Content, и т.д.)
- Все страницы (DashboardPage, UsersPage, и т.д.)
- Все вложенные элементы

Для каждого компонента извлечь:
- **Название** (из data-name или id)
- **Размеры**: width, height (в px)
- **Позиция**: x, y (в px)
- **Цвета**: 
  - background-color (HEX)
  - color (HEX)
  - border-color (HEX)
- **Отступы**: padding, margin (в px)
- **Границы**: border-width, border-radius (в px)
- **Шрифты**: font-size, font-weight, font-family
- **Тени**: box-shadow
- **Структура**: вложенность, flex/grid параметры

**Список компонентов для анализа** (из pyramid-rigadev-top):
1. Sidebar (78:234)
2. TopBar (78:297)
3. Main Content Container (78:296)
4. DashboardPage (78:324)
5. Все другие компоненты, найденные в файле

#### 2.3 Сравнение с текущей реализацией
- Прочитать текущие файлы компонентов
- Прочитать текущие CSS файлы
- Прочитать variables.css
- Выявить различия

## 3. Компоненты для изменения

### 3.1 Sidebar
**Файл**: `src/components/layout/Sidebar.jsx`, `Sidebar.css`

**Изменения**:
1. Фон: изменить с `#FFFFFF` на `#f2f1ec`

   - Селектор: `.sidebar`
   - Свойство: `background-color`
   - Старое значение: `var(--card-bg-color)` → `#FFFFFF`
   - Новое значение: `var(--color-background-sidebar)` → `#f2f1ec`

2. Активный элемент Dashboard:
   - Файл: `Sidebar.css`
   - Селектор: `.sidebar__nav-item--active`
   - Изменения:
     - `background-color`: `#efff42`
     - `border-radius`: `4px 4px 0 0`

## Критерии качества работы Orchestrator

### ✅ Обязательно
- Все компоненты из Figma проанализированы
- Все цвета в HEX формате
- Все размеры в px
- Четкие инструкции для Developer
- Приоритеты расставлены

### ❌ Недопустимо
- Пропущенные компоненты
- Неточные цвета (без HEX)
- Размытые инструкции
- Отсутствие приоритетов
- Нет критериев приемки

## Процесс анализа всего файла

### Шаг 1: Получить структуру файла
```
1. Вызвать mcp0_get_design_context без nodeId
2. Получить XML структуру всего файла
3. Определить все основные компоненты по их ID
```

### Шаг 2: Анализ каждого компонента
```
Для каждого найденного компонента:
1. Вызвать mcp0_get_design_context с конкретным nodeId
2. Извлечь все стили (цвета, размеры, отступы)
3. Записать в спецификацию
```

### Шаг 3: Создание полной спецификации
```
Спецификация должна включать:
- Все компоненты из файла (не только Sidebar!)
- Все цвета из всех компонентов
- Все размеры и отступы
- Все структурные изменения
```
