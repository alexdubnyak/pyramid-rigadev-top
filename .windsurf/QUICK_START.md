# 🚀 Quick Start - Figma Implementation Workflow

## 🔄 Самый быстрый способ (рекомендуется)

**Просто введите:**
```
/loop
```

Это запустит полный автоматический цикл: Orchestrator → Developer → QA → Orchestrator (решение).

Все роли выполнятся автоматически, вы просто будете наблюдать за процессом и получите финальный результат.

**Время:** 25-80 минут  
**Итерации:** автоматически (максимум 3)  
**Результат:** готовый код + отчеты

---

## 📝 Пошаговый запуск (для контроля каждого шага)

### Шаг 1: Orchestrator (Анализ и постановка задачи)

**Скопируйте и вставьте в чат:**
```
Я - Orchestrator. Моя задача:
1. Получить дизайн из Figma через mcp0_get_design_context
2. Проанализировать все компоненты, цвета, размеры, отступы
3. Сравнить с текущей реализацией
4. Создать детальную спецификацию в /.windsurf/figma-spec.md
5. Передать задачу Developer

Начинаю работу с анализа макета Figma...
```

**Ожидаемый результат:**
- Создан файл `/.windsurf/figma-spec.md`
- Сообщение: "Спецификация готова. Передаю задачу Developer."

---

### Шаг 2: Developer (Реализация)

**Скопируйте и вставьте в чат:**
```
Я - Developer. Моя задача:
1. Прочитать спецификацию из /.windsurf/figma-spec.md
2. Реализовать все изменения согласно приоритетам (P0 → P1 → P2)
3. Использовать CSS переменные
4. Сохранить существующие паттерны
5. Создать детальный отчет в /.windsurf/dev-report.md
6. Передать результат QA

Начинаю работу с чтения спецификации...
```

**Ожидаемый результат:**
- Изменены файлы согласно спецификации
- Создан файл `/.windsurf/dev-report.md`
- Сообщение: "Разработка завершена. Передаю результат QA."

---

### Шаг 3: QA (Тестирование)

**Скопируйте и вставьте в чат:**
```
Я - QA. Моя задача:
1. Прочитать спецификацию из /.windsurf/figma-spec.md
2. Прочитать отчет Developer из /.windsurf/dev-report.md
3. Получить эталонные значения из Figma
4. Проверить каждый компонент на соответствие
5. Создать детальный отчет в /.windsurf/qa-report.md
6. Передать результат Orchestrator

Начинаю работу с изучения документации...
```

**Ожидаемый результат:**
- Создан файл `/.windsurf/qa-report.md`
- Сообщение: "Тестирование завершено. Передаю результат Orchestrator."
- Статус: PASS или FAIL

---

### Шаг 4: Orchestrator (Принятие решения)

**Скопируйте и вставьте в чат:**
```
Я - Orchestrator. Моя задача:
1. Прочитать QA отчет из /.windsurf/qa-report.md
2. Принять решение: завершить или начать новую итерацию
3. Если есть несоответствия - создать новую спецификацию
4. Если все хорошо - завершить workflow

Начинаю работу с анализа QA отчета...
```

**Возможные результаты:**

**Если PASS:**
- Создан файл `/.windsurf/implementation-complete.md`
- Сообщение: "Внедрение макета завершено успешно!"

**Если FAIL:**
- Обновлен файл `/.windsurf/figma-spec.md` (итерация 2)
- Сообщение: "Требуется новая итерация. Передаю задачу Developer."
- Вернуться к **Шагу 2**

## Структура файлов

После выполнения workflow у вас будут созданы:

```
.windsurf/
├── workflows/
│   └── figma-implementation.md       # Описание workflow
├── roles/
│   ├── orchestrator.md               # Спецификация роли Orchestrator
│   ├── developer.md                  # Спецификация роли Developer
│   └── qa.md                         # Спецификация роли QA
├── figma-spec.md                     # Спецификация изменений (создается Orchestrator)
├── dev-report.md                     # Отчет разработчика (создается Developer)
├── qa-report.md                      # Отчет тестировщика (создается QA)
└── implementation-complete.md        # Финальный отчет (создается Orchestrator при PASS)
```


## Критерии успеха

### ✅ Workflow успешен если:
- QA отчет показывает статус PASS
- Процент соответствия ≥ 95%
- Нет критичных несоответствий
- Создан файл implementation-complete.md

### ❌ Требуется новая итерация если:
- QA отчет показывает статус FAIL
- Процент соответствия < 95%
- Есть критичные несоответствия (P0)

## Советы

1. **Не прерывайте роли** - дайте каждой роли завершить работу
2. **Читайте отчеты** - они содержат важную информацию
3. **Следите за итерациями** - обычно достаточно 1-2 итераций
4. **Проверяйте в браузере** - визуальная проверка важна
5. **Сохраняйте отчеты** - они помогут в будущем

## Troubleshooting

### Проблема: Orchestrator не может получить дизайн из Figma
**Решение**: Убедитесь что Figma файл открыт в Figma Desktop App

### Проблема: Developer не находит файлы
**Решение**: Проверьте пути в спецификации

### Проблема: QA показывает много несоответствий
**Решение**: Это нормально для первой итерации, запустите новую итерацию

### Проблема: Workflow зациклился
**Решение**: Прочитайте отчеты и вручную исправьте критичные проблемы
