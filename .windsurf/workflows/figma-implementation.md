# Figma Implementation Workflow

Этот workflow использует три роли для точного внедрения макета из Figma:
- **Orchestrator** - анализирует макет и ставит задачи
- **Developer** - реализует изменения
- **QA** - проверяет соответствие макету

## Роли и их обязанности

### 🎯 Orchestrator (Оркестратор)
**Цель**: Анализ макета Figma и постановка четких задач разработчику

**Обязанности**:
1. Получить дизайн из Figma через `mcp0_get_design_context`
2. Проанализировать все компоненты, цвета, размеры, отступы
3. Создать детальную спецификацию изменений
4. Передать задачу Developer
5. Принять результат от QA и принять решение о завершении или новой итерации

**Выходные данные**:
- Файл спецификации `/.windsurf/figma-spec.md` с:
  - Список компонентов для изменения
  - Точные значения цветов (HEX)
  - Размеры и отступы (px)
  - Структурные изменения
  - Приоритет изменений

---

### 💻 Developer (Разработчик)
**Цель**: Реализовать изменения согласно спецификации

**Обязанности**:
1. Прочитать спецификацию из `/.windsurf/figma-spec.md`
2. Внести изменения в код (CSS, JSX)
3. Обновить переменные в `variables.css`
4. Обновить компоненты и стили
5. Создать отчет о выполненных изменениях
6. Передать результат QA

**Выходные данные**:
- Файл отчета `/.windsurf/dev-report.md` с:
  - Список измененных файлов
  - Описание каждого изменения
  - Скриншоты (если возможно)

---

### ✅ QA (Тестировщик)
**Цель**: Проверить соответствие реализации макету Figma

**Обязанности**:
1. Прочитать спецификацию из `/.windsurf/figma-spec.md`
2. Прочитать отчет разработчика из `/.windsurf/dev-report.md`
3. Сравнить реализацию с макетом Figma
4. Проверить все измененные компоненты
5. Создать отчет о найденных несоответствиях
6. Передать результат Orchestrator

**Выходные данные**:
- Файл отчета `/.windsurf/qa-report.md` с:
  - Статус проверки (PASS/FAIL)
  - Список несоответствий (если есть)
  - Рекомендации по исправлению
  - Процент соответствия макету

## Workflow Loop

### Итерация 1: Orchestrator → Developer

**Шаг 1**: Запустите роль Orchestrator
```
Я - Orchestrator. Моя задача:
1. Получить дизайн из Figma
2. Проанализировать все компоненты
3. Создать детальную спецификацию в /.windsurf/figma-spec.md
4. Передать задачу Developer

Начинаю работу...
```

---

### Итерация 2: Developer → QA

**Шаг 2**: Запустите роль Developer
```
Я - Developer. Моя задача:
1. Прочитать спецификацию из /.windsurf/figma-spec.md
2. Реализовать все изменения
3. Создать отчет в /.windsurf/dev-report.md
4. Передать результат QA

Начинаю работу...
```

---

### Итерация 3: QA → Orchestrator

**Шаг 3**: Запустите роль QA
```
Я - QA. Моя задача:
1. Прочитать спецификацию из /.windsurf/figma-spec.md
2. Прочитать отчет из /.windsurf/dev-report.md
3. Проверить соответствие макету
4. Создать отчет в /.windsurf/qa-report.md
5. Передать результат Orchestrator

Начинаю работу...
```

---

### Итерация 4: Orchestrator - Финальное решение

**Шаг 4**: Запустите роль Orchestrator для принятия решения
```
Я - Orchestrator. Моя задача:
1. Прочитать QA отчет из /.windsurf/qa-report.md
2. Принять решение: завершить или начать новую итерацию
3. Если есть несоответствия - создать новую спецификацию
4. Если все хорошо - завершить workflow

Начинаю работу...
```

---

## Как использовать

1. **Скопируйте и вставьте** текст из "Шаг 1" в чат
2. **Дождитесь завершения** работы Orchestrator
3. **Скопируйте и вставьте** текст из "Шаг 2" в чат
4. **Дождитесь завершения** работы Developer
5. **Скопируйте и вставьте** текст из "Шаг 3" в чат
6. **Дождитесь завершения** работы QA
7. **Скопируйте и вставьте** текст из "Шаг 4" в чат
8. **Повторяйте цикл** пока Orchestrator не подтвердит завершение

## Критерии качества

### Для Orchestrator:
- ✅ Все компоненты из Figma проанализированы
- ✅ Все цвета указаны в HEX формате
- ✅ Все размеры указаны в px
- ✅ Приоритеты расставлены

### Для Developer:
- ✅ Все изменения из спецификации реализованы
- ✅ Код соответствует существующим паттернам
- ✅ Нет ошибок в консоли
- ✅ Отчет содержит все измененные файлы

### Для QA:
- ✅ Проверены все компоненты из спецификации
- ✅ Цвета совпадают с макетом (допуск ±1 в HEX)
- ✅ Размеры совпадают с макетом (допуск ±2px)
- ✅ Структура компонентов соответствует макету
- ✅ Отчет содержит конкретные несоответствия

---

## Примечания

- Иконки не меняются (в Figma только плейсхолдеры)
- Все изменения должны быть обратимы
- Сохранять существующую функциональность
- Использовать CSS переменные где возможно
