# Figma Implementation Workflow System

Система для точного внедрения макетов из Figma с использованием трех специализированных ролей.

## 📋 Обзор

Эта система автоматизирует процесс внедрения дизайна из Figma в код, используя три роли:

1. **🎯 Orchestrator** - Анализирует макет и создает детальную спецификацию
2. **💻 Developer** - Реализует изменения согласно спецификации
3. **✅ QA** - Проверяет соответствие реализации макету

## 🎯 Роли

### Orchestrator
- **Вход**: Figma файл
- **Выход**: `figma-spec.md`
- **Задача**: Создать детальную спецификацию изменений
- **Детали**: [roles/orchestrator.md](./roles/orchestrator.md)

### Developer
- **Вход**: `figma-spec.md`
- **Выход**: `dev-report.md` + измененный код
- **Задача**: Реализовать изменения согласно спецификации
- **Детали**: [roles/developer.md](./roles/developer.md)

### QA
- **Вход**: `figma-spec.md`, `dev-report.md`, Figma файл
- **Выход**: `qa-report.md`
- **Задача**: Проверить соответствие реализации макету
- **Детали**: [roles/qa.md](./roles/qa.md)

## 📊 Метрики качества

### Orchestrator
- ✅ Все компоненты проанализированы
- ✅ Все цвета в HEX формате
- ✅ Все размеры в px
- ✅ Приоритеты расставлены

### Developer
- ✅ Все изменения реализованы
- ✅ Используются CSS переменные
- ✅ Код соответствует паттернам
- ✅ Нет ошибок

### QA
- ✅ Цвета совпадают (допуск ±1 в HEX)
- ✅ Размеры совпадают (допуск ±2px)
- ✅ Структура соответствует
- ✅ Общее соответствие ≥ 95%

## 🎓 Примеры использования

### Пример 1: Первое внедрение макета
1. Запустить Orchestrator
2. Дождаться создания спецификации
3. Запустить Developer
4. Дождаться реализации
5. Запустить QA
6. Если PASS - готово, если FAIL - новая итерация

### Пример 2: Обновление существующего дизайна
1. Запустить Orchestrator (он сравнит с текущей реализацией)
2. Получить спецификацию только с изменениями
3. Запустить Developer для реализации
4. Запустить QA для проверки
5. Принять решение

### Пример 3: Исправление несоответствий
1. Прочитать qa-report.md
2. Запустить Orchestrator с фокусом на несоответствия
3. Запустить Developer для исправлений
4. Запустить QA для повторной проверки
5. Принять решение

## 🔧 Настройка

### Допуски
Можно изменить в спецификациях ролей:
- Цвета: ±1 в HEX (по умолчанию)
- Размеры: ±2px (по умолчанию)
- Общее соответствие: ≥95% (по умолчанию)

### Приоритеты
- **P0** (Критично) - блокирует приемку
- **P1** (Высокий) - важно, но не критично
- **P2** (Средний) - можно отложить

## 📝 Лучшие практики

### Для Orchestrator
1. Получать дизайн по частям если он большой
2. Сравнивать с текущей реализацией
3. Расставлять приоритеты
4. Создавать четкие инструкции

### Для Developer
1. Следовать приоритетам (P0 → P1 → P2)
2. Использовать CSS переменные
3. Сохранять паттерны проекта
4. Документировать изменения

### Для QA
1. Получать точные значения из Figma
2. Проверять каждый компонент
3. Документировать несоответствия
4. Давать конкретные рекомендации

## 🐛 Troubleshooting

### Orchestrator не может получить дизайн
- Убедитесь что Figma Desktop App запущен
- Проверьте что файл открыт
- Попробуйте получить дизайн по частям

### Developer не находит файлы
- Проверьте пути в спецификации
- Убедитесь что файлы существуют
- Проверьте права доступа

### QA показывает много несоответствий
- Это нормально для первой итерации
- Запустите новую итерацию
- Проверьте критичные несоответствия

### Workflow зациклился
- Прочитайте все отчеты
- Найдите повторяющиеся проблемы
- Исправьте вручную если нужно
